# 架构说明

> 三个定位文件已经分别定义了管家、个人知识库、库skill的角色。
> 这个文件说它们怎么结合。

## 三层架构

```
┌─────────────────────────────────────────┐
│                管家                       │
│  调度层：跟用户对话、引导任务、调用库skill   │
│  有完整读写权限，修改内容推荐走库skill       │
└──────────────────┬──────────────────────┘
                   │ 调用
                   ▼
┌─────────────────────────────────────────┐
│      个人知识库（自带库skill）              │
│  数据层 + 维护工具                         │
│  库skill：初始化 / 文档更新 / 归档 / 体检 / 迁移 │
└──────────────────┬──────────────────────┘
                   │ 存储
                   ▼
┌─────────────────────────────────────────┐
│     项目数据（文档 + 源代码 + 工作区）       │
└─────────────────────────────────────────┘
```

## 设计原则：数据驱动，而非约定驱动

系统不预设文件一定存在。管家和库skill 在操作时先检查目标文件是否存在：

- **存在** → 读/更新它
- **不存在** → 跳过，或按需创建，或告知用户让用户决定

所有流程的默认行为是**适应**，不是**报错**。文件丢了、格式不对、某个索引还没有创建——这些都不影响系统的其余部分正常工作。

这条原则贯穿整个系统——入口判断工作区存在与否走不同路径，跨项目索引不存在时扫目录临时替代，库skill 的写入目标不存在时首次创建。


## 数据流转

1. 用户跟管家对话，讨论计划
2. 管家读写 .管家/ 工作区，记录进度
3. 用户需要查资料 → 管家直接读知识库
4. 用户需要整理/修改数据 → 管家调用库skill（比手写更方便、更规范）
5. 库skill 执行具体工作，操作知识库
6. 紧急情况下管家也可以直接写，但修改数据内容优先走库skill

## 三个文件读哪个

- 想知道"系统是什么" → 读 系统介绍
- 想知道"某个角色做什么" → 读对应的定位文件
- 想知道"怎么配合" → 读这个架构说明
