# 项目内核

## 这个项目解决什么问题

你用 AI 写代码。AI 帮你干活，但没有人帮你管"你"。

你手上同时跑着好几个任务，每个做到哪一步了、卡在什么地方、哪些文档该更新了——这些事你只能自己记。起一个新任务，先得跟 AI 交代一遍背景、技术栈、要做到什么效果。做一个模块做完了，文档还停在旧版本，下次再聊 AI 完全不知道代码已经变了。同时跑着智能锁开发、面试复习、一个新方案的调研，上午写这个下午看那个——每次切换，重新进入状态的成本高得离谱。

这些都不是你做事的核心——但没有人帮你管这些事，你只能自己扛。

## 核心：一个全职秘书

管家和知识库只是当前的实现方式。如果明天有更好的方案，它们都可以被替换。

真正不变的是这个角色——

一个秘书，对你所有的项目和任务心里有数。它的工作方式：

```
你有想法 → 它帮你理清技术链路，给出建议，等你拍板
你确定了 → 它生成文档骨架，一步步引导你执行
你干活   → 它在旁边记进度、记偏好、记你踩过的坑
你做完了 → 它提醒你更新文档，帮你收尾归档
你切换   → 它知道你另一个任务做到哪了，无缝接上
你回来   → 你问"昨天做到哪了"，它告诉你
```

它不替你决策——技术选型你定。它不未经讨论就写代码——先把底层逻辑讲清楚。它不未经确认就操作文件——你点头它才动手。

但它也不只是传话筒。你问它一个问题，它给你分析、给你建议，然后等你拍板。就像一个称职的秘书：把信息理好、把选项摆出来、跟你说哪个更好——但决定始终是你做。

## 为什么不是"又一个 skill 模板"

拆开来看，管家系统就三样东西：

- **库 skill** — 蒸馏出来的工作流。任务初始化、文档更新、归档、体检，本质上是把靠谱的工作习惯写成可复用的步骤。
- **管家** — 一个调度角色。判断什么时候该做什么、为什么做，给建议但不替用户决定。
- **偏好记录** — 用户用过什么、偏好像什么、踩过什么坑，记在 .管家/ 下。

单看哪一样，市面上都有。skill 模板有人做，AI 记忆有人做，角色扮演也有人做。

**但没人把它们串成一个闭环。**

```
你用了一次 → 管家发现你偏好某个做法 → 记下来
下一次 → 管家读偏好 → 主动建议"上次你不是喜欢 X 吗，这次还用 X？"
你改了主意 → 管家更新偏好 → 下次基于新的偏好调整
```

三个东西单看都不新鲜。串成一个越用越懂你的反馈循环——市面上确实没有。

## 跟现有方案的区别

市面上大多数方案解决的是"帮 AI 记住项目上下文"——让 AI 更聪明、更有记忆。

我们解决的是"帮人管理自己的工作状态"——让你不用自己记进度、记决策、记下一步该干什么。AI 可以换，项目可以换，但这个秘书始终跟着你。

跨项目、跨任务的调度和管理，是目前没有人完整做过的方向。

## 一句话内核

**一个替你管理所有工作状态的全职秘书。你决策，它执行。**
