# 项目内核

## 这个项目解决什么问题

所有深度使用 AI 来写代码的人，都有同一个困扰——

你手上的项目越来越多，每个项目都需要一套 MD 文档：项目背景、技术栈、架构、进度记录、踩坑日志。这些文档是 AI 能帮你干活的前提，但管理它们本身就是一种负担。

起一个新项目 -> 先写 CLAUDE.md、CONTEXT.md、REQUIREMENTS.md……代码还没写，文档先写累了。
做完一个项目 -> 要归档、总结、更新索引，不然以后找不回来。
同时跑多个项目 -> 每个项目独立的文件、独立的进度、独立的上下文。切换一次，重新进入状态的成本高得离谱。
换一个 AI 工具 -> 又要重新交代一遍你是谁、你在做什么。

这些都不是你做事的核心——它们是你为了“让 AI 能帮你干活”而不得不做的杂事。但没有人帮你管这些事，你只能自己扛。

## 核心不是管家，也不是知识库

管家和知识库只是我们当前的实现方式。如果明天有更好的方案，它们都可以被替换。

```
第一代：管家 + MD 知识库
第二代：更好的管家 + 更好的知识库管理
第三代：……
```

真正不变的是这个工作流——

```
你跟管家讨论一个任务的计划
  -> 管家把计划拆成一步步的方向
  -> 你执行，管家在旁边支持你
  -> 做完，管家帮你记录进度、整理产出
  -> 换一个任务，管家帮你接上
  -> 下次回来，管家知道你在哪
```

不管管家是谁、知识库用什么存，这个工作流是核心。

## 跟现有方案的区别

现在已经有 project-butler、aethel、spec-driven-workflow 等项目在解决类似的问题。

但它们大多解决的是“一个项目的记忆”——帮 AI 记住当前这个项目做到哪了。

我们的方案解决的是“一个人的所有项目和任务”——不管你有几个任务在跑、它们怎么切换、你用什么工具，总有一个管家知道你全部的状态。

跨项目、跨任务的调度，是目前没有人完整做过的方向。

## 一句话内核

**让 AI 帮你干活的人，也应该有一个 AI 来帮他们管理“让 AI 干活”这件事。**
