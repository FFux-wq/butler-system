# 任务初始化

> 库skill 之一。当用户说「我想做一件事」「起个新任务」「新建一个项目」「有个想法」时，管家触发此流程。
> 无论是什么类型的任务——做产品、学知识、整理资料、方案预研、个人计划——都从这里开始。

## 使用原则

以下流程是参考框架，不是剧本。你跟用户的对话是活的——

- **已经聊到的直接跳过。** 对话中自然覆盖的内容，说一句"刚才我们讨论了XX，这部分就算过了"，然后继续。不要倒回去重问。
- **提问用已有的信息。** 用户已经说过的东西就别再问——你的问题是用来补全缺失的信息，不是用来确认已知的事实。
- **先盘再补。** 每进入一个新步骤，先回顾对话里有没有覆盖这个点的内容。有的就总结确认，缺的才问。

## 工作方式

管家读到这个文件后，**切换到任务初始化模式**，按以下流程执行。

## 流程总览

```
确定任务类型 → 聊清楚效果 → 梳理框架（由粗到细，层层下钻） → 拟定文档清单 → 生成骨架
```

整个流程只有一条主线：从最粗的框架开始，一层层往下钻。钻的过程中，技术选型自然浮出来，风险点自然暴露。钻到所有东西都清楚了，才进入文档生成。

---

## 通用规则

以下规则适用于所有任务类型和所有分支。每个分支不再重复写。

### 一、先拼再问

用户说的信息可能是碎的——东一句西一句，想到哪说到哪。你的任务是**把他说的碎片收起来，拼成一张完整的图，详细讲给他听让他确认**。不是原样复读他说的话——是把散的理顺、缺的标出来、断的接上。拼完之后他确认了，再往下走。

缺的地方，只问"框架链条上断掉的地方"——不问技术参数和实现细节，那些在下一层自然会覆盖到。

### 二、层层下钻，由粗到细

不管什么场景，都先把这一层的大框架聊清楚，框架定了再往下钻。每一层都是同一个逻辑。

**钻到什么程度停？** 不是钻到每一个电阻型号或每一行代码。判断标准：**这一个环节，清楚到"你能动手做"或"你能交代给别人做"的程度，就停了。** 如果用户觉得"剩下的执行的时候再说"，也停。

### 三、摊出组织原则

每次做分类、分阶段、拆模块时，**必须说清楚"凭什么这么分"**——是按数据流（从采集到展示）、按工作性质（硬件 / 软件 / 前端）、按技术栈、还是按交付节奏。把它摊出来，用户一眼就能看出分歪了没有。

### 四、每层复查 + 中途落地

每一层框架聊完，做复查收口——收拢这一层聊到的技术选型、风险点、前置依赖。不重新问，是把聊到的东西抓出来整理好。

**关键：复查结果当场写进文件，不要全存脑子里。** 在任务目录下维护一个 `当前状态.md`，记录：

- 当前钻到哪个阶段、哪一层
- 每层复查的收拢结果
- 待确认的事项

管家每次继续往下钻之前，先读这个文件，不用翻几十轮对话。总收口时也从这里汇总，不是从记忆里拼。

### 五、用户喊停的出口

用户说"够了先这样""剩下的到时候再说"时：

1. 当前这一层如果已经聊了一些内容 → 做一个快速收口（三句话：链到哪了、哪些定了、哪些待定）
2. 把当前钻到的位置记录到 `当前状态.md`
3. 跳到拟定文档清单

不强行钻完。

### 六、递归目录 + 分层生成

目录结构不是固定的两层，是递归的。每个"阶段目录"如果太复杂需要再拆，就在那个目录下再跑一遍同样的流程，生成同样的子目录结构。每一层自己管自己——任务初始化 skill 的流程在每一层都是同一套逻辑，不因层级不同而改变。

**文档分两层生成，不一次性全部生成：**

- **任务级文档** — 整体框架确认后立即生成。包括：任务介绍.md（整体链路、阶段划分、全局技术栈）、开发计划.md（所有阶段的概要——每个阶段一句话目标 + 一句话交付）。这些是"地图"，框架定了就能写。
- **阶段级文档** — 每个阶段启动时再生成。包括：阶段介绍.md、阶段进度.md、该阶段详细的技术决策和风险点。这些是"施工图"，等到真要动手那个阶段的时候再写，那时候上下文是新鲜的，写出来的东西最准确。

## 第一步：确定任务类型

先问用户：
**"这是一个什么类型的任务？"**

- 🛠️ **工程项目** — 做一个产品、写一段代码
- 📖 **学习/复习** — 学一个知识点、复习项目内容、准备面试
- 📂 **文档整理** — 整理笔记、归档资料、归纳总结
- 🧩 **方案预研** — 对比方案、技术选型、调研评估
- 🎯 **个人计划** — 训练计划、读书计划、阶段性目标

用户回答后进入对应分支。

---

## 工程项目分支

工程项目的核心是**流程链**——数据从哪来、经过什么、到哪去。所有的讨论都围绕这条链展开。

### 第二步：聊清楚效果

见通用规则一（先拼再问）。用一段话把从开始到结束的完整场景串起来给他确认。这一步的任务是把"想法"变成"一个完整的画面"。

这一步不碰技术参数和实现细节——那些在梳理流程链时自然会覆盖到。

### 第三步：梳理流程链（层层下钻）

#### 第一层：粗链

见通用规则一（先拼再问）。用户给的可能是"GPS 到 MCU 到 Flash 到 WiFi 到服务器"这种碎片——帮他拼成一条完整的链路：每个环节之间怎么连、数据怎么流、从起点到终点经过什么。缺的环节标出来让他补，断的地方指出来让他确认。

讲完之后，判断这条路能不能通、有没有明显断掉的地方。如果链路上有不踏实的地方，指出来让用户确认。

用户确认粗链后，判断需不需要分阶段：

"这条路看起来可以一口气走完 / 比较长，建议分几个阶段。你怎么看？"

管家自己判断要不要分阶段、分几个、按什么原则分。用户只需要确认或调整。不把决策压力甩回给用户。

分阶段时，见通用规则三（摊出组织原则）——必须说清楚"凭什么这么分"，用户确认后再往下。

#### 第二层及以后：逐阶段细化

见通用规则二（层层下钻）、四（复查+落地）、五（喊停出口）。

**关键节奏**：用户确认阶段划分后，**先把所有阶段的第二层链都拉出来**（同一层级，先框架），让用户看到全局的阶段间链路，然后再逐阶段往下钻细节。不要钻完阶段一的所有层级再去看阶段二——那样用户会忘了后面还有什么。

每个阶段钻完之后，复查结果当场写入 `当前状态.md`（见通用规则四）。

**管家的角色**：管家不是在审问用户"这个用什么技术""那个有什么风险"。管家是**用自己的技术知识帮用户一起看这条链**——哪些环节常规做法是什么、哪些地方可能有坑、哪些环节之间有依赖。管家主动给判断，不是把球踢回去让用户自己想。

#### 全部钻完之后：总收口

所有阶段都钻完了，从 `当前状态.md` 汇总（不是从记忆里拼），做最终总结呈现给用户：

"好，整体梳理完了。最后的总结：

你要做的是：[效果一句话]

分 N 个阶段：
- 阶段一：[目标]，[关键交付]，[技术栈]，[风险]
- 阶段二：[目标]，[关键交付]，[技术栈]，[风险]
- ...

全局技术栈：[所有阶段技术选型合并]
全局风险点：[所有阶段风险点合并]
全局依赖：[所有阶段前置依赖合并]

以上是所有聊到的东西。你看有没有漏的、不对的？"

用户确认后，按通用规则六（分层生成）生成任务级文档。阶段级文档留到各自启动时再生成。

## 学习/复习分支

学习/复习的核心是**知识框架**——从哪开始、分几个部分、各部分之间的依赖关系。

### 第二步：聊清楚效果

见通用规则一（先拼再问）。用户说"我要复习操作系统"——帮他拼一个画面：复习的范围大概是什么、目标是搞懂原理还是能复现还是能讲清楚、大概分几个部分。拼出来给他确认。

### 第三步：梳理复习框架

根据复习目标和范围，跟用户一起理一条复习路线：

- 从哪开始？分几个部分？
- 每个部分大概覆盖什么内容？
- 各部分之间有没有前置依赖？（学 B 之前需要先懂 A？）

不需要排到天——大框架定了，具体每天学什么，执行的时候再说。

框架聊完之后，见通用规则四（复查+落地）：收拢复习框架、各部分依赖关系，写入 `当前状态.md`。

用户确认后，按通用规则六（分层生成）生成文档。

---

## 文档整理分支

文档整理的核心是**分类框架**——按什么分、分几类。

### 第二步：聊清楚效果

见通用规则一（先拼再问）。用户说"帮我把笔记整理一下"——帮他拼一个画面：整理的范围是什么、按什么分类、整理成什么样算完成。拼出来给他确认。

### 第三步：梳理分类框架

见通用规则一（先拼再问）。跟用户一起定分类框架——按时间、按技术栈、按项目？分几个大类？不需要逐文件列清单，分类框架定了就行。

框架聊完之后，见通用规则四（复查+落地）：收拢分类框架，写入 `当前状态.md`。

用户确认后，按通用规则六（分层生成）生成文档。

---

## 方案预研分支

方案预研的核心是**候选对比**——比什么、怎么比。

### 第二步：聊清楚效果

见通用规则一（先拼再问）。用户说"我想看看这几个方案哪个好"——帮他拼一个画面：研究的目标是什么、关注哪些维度、研究到什么程度算完成。拼出来给他确认。

### 第三步：确定候选和对比框架

跟用户一起定：
- 候选清单（用户有明确的就记下来，只有一个方向就帮补充建议）
- 对比维度（性能、成本、生态、功耗、开发难度……按用户关注的来）

框架聊完之后，见通用规则四（复查+落地）：收拢候选清单和对比维度，写入 `当前状态.md`。

用户确认后，按通用规则六（分层生成）生成文档。

---

## 个人计划分支

个人计划的核心是**里程碑**——周期多长、分几个阶段、每个阶段的目标。

### 第二步：聊清楚效果

见通用规则一（先拼再问）。用户说"我想做一个训练计划"——帮他拼一个画面：什么计划、周期多长、大致分几个阶段、怎么算完成。拼出来给他确认。

### 第三步：梳理计划框架

跟用户一起定大框架：分几个阶段（每周？每月？按里程碑？），每个阶段的大致目标是什么。不需要细化到每天——框架定了，具体执行的时候再排。

框架聊完之后，见通用规则四（复查+落地）：收拢计划框架和各阶段目标，写入 `当前状态.md`。

用户确认后，按通用规则六（分层生成）生成文档。

---

## 拟定文档清单

读 `库级md文件/库文件模板参考.md`，根据前面收集到的信息，判断本次任务适合哪些文件类型，列出建议清单给用户确认。

判断依据：
- **任务类型**：工程项目 vs 学习复习 vs 文档整理 vs 方案预研 vs 个人计划
- **任务复杂度**：简单任务可以减少文件，复杂任务需要增加
- **用户的习惯**：如果用户之前喜欢写踩坑记录就保留，否则跳过

清单列出后，问用户：
- "我建议生成以下 N 个文件：[列表]。你觉得哪些要、哪些不要？有没有其他想加的？"
- 用户确认后按确认清单生成。

---

## 生成 MD 文档

根据确认的文档清单和阶段划分，生成对应的骨架文件。

所有任务类型都包含 `当前状态.md`（见通用规则四），记录管家当前所处阶段、层级和待确认事项。

工程项目的典型结构（供参考）：

```
任务目录/
|-- 当前状态.md        - 管家维护：当前钻到的位置、复查收拢结果、待确认事项
|-- 任务介绍.md        - 任务名称、目标、效果描述、流程链、技术栈
|-- 开发计划.md        - 所有阶段的概要（每阶段一句话目标+一句话交付）
|-- 学习记录.md        - 空模板
|-- 踩坑记录.md        - 空模板（如果需要）
|-- 技术决策.md        - 空模板（如果需要）

|-- 阶段一_XXX/        （分阶段时启用，阶段启动时生成）
|   |-- 阶段介绍.md
|   |-- 阶段进度.md

|-- 阶段二_XXX/        （以此类推）
```

学习/复习任务的典型结构（供参考）：

```
任务目录/
|-- 当前状态.md
|-- 任务介绍.md        - 复习目标、范围
|-- 学习笔记.md        - 学到的东西、重点、疑问
|-- 进度跟踪.md        - Checklist 跟踪
```

方案预研任务的典型结构（供参考）：

```
任务目录/
|-- 当前状态.md
|-- 任务介绍.md        - 预研目标、候选清单、关注维度
|-- 方案对比.md        - 各候选维度的详细对比
|-- 推荐方案.md        - 结论和理由

|-- 阶段一_调研/       （分阶段时启用，阶段启动时生成）
|-- 阶段二_验证/
```

个人计划的典型结构（供参考）：

```
任务目录/
|-- 当前状态.md
|-- 任务介绍.md        - 计划目标、周期、完成标准
|-- 进度跟踪.md        - 各阶段的完成情况

|-- 阶段一_XXX/
|-- 阶段二_XXX/
```

每个文件写上标题和基本结构。内容不填满，留出骨架让用户后续补充。

目录结构是递归的（见通用规则六）。阶段目录本身如果太复杂，就在那个目录下再跑一遍同样的流程，生成同样的子目录结构。

---

## 完成后

生成完毕后告诉用户：
"任务已创建，在 XXX 路径下。你现在可以：跟管家讨论第一步做什么、直接开始执行、或者随时说'换一个任务'切换。"

然后退出任务初始化模式，返回管家状态。
