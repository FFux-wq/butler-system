# 任务初始化

> 库skill 之一。当用户说「我想做一件事」「起个新任务」「新建一个项目」「有个想法」时，管家触发此流程。
> 无论是什么类型的任务——做产品、学知识、整理资料、方案预研、个人计划——都从这里开始。

## 工作方式

管家读到这个文件后，**切换到任务初始化模式**，按以下步骤逐一执行。每一步做完再走下一步。全部完成后退出模式，返回管家状态。

## 流程总览

```
确定任务类型 → 问清楚内容 → 验证技术路线 → 预判风险难点 → 确定是否分阶段 → 技术选型 → 拟定文档清单 → 生成骨架
```

---

### 第一步：确定任务类型

先问用户：
**"这是一个什么类型的任务？"**

- 🛠️ **工程项目** — 做一个产品、写一段代码
- 📖 **学习/复习** — 学一个知识点、复习项目内容、准备面试
- 📂 **文档整理** — 整理笔记、归档资料、归纳总结
- 🧩 **方案预研** — 对比方案、技术选型、调研评估
- 🎯 **个人计划** — 训练计划、读书计划、阶段性目标

用户回答后进入对应分支。

---

## 工程项目分支

### 第二步：问清楚想要什么效果

问用户：
- **"你想做一个什么东西？最终用起来是什么感觉？"**

帮他展开画面：
- 谁来用这个？怎么用？
- 最终用户看到什么、操作什么？
- 能解决什么问题或带来什么体验？

用户回答后，复述确认。

### 第三步：理清顶层架构

- 分几个模块来实现这个效果？
- 数据链路是怎样的？

用户回答后，复述确认。

### 第四步：确认流程链

复述前面收集到的信息（最终效果 + 模块划分 + 数据链路），确认无误。
**这一步只确认，不问新技术细节。** 技术选型留在后面。

确认后进入下一步：验证技术路线的可行性。

### 第五步：技术路线连通性验证

在深入讨论具体技术选型之前，先做一个**全链路扫描**，确保从头到尾技术路线是通的。

基于前面确认的模块划分和数据链路，管家**用自己的技术知识**逐段审视。用户只有一个想法，不一定清楚每个环节的技术细节——管家的任务是用自己的知识帮他补全和验证。

按数据流向逐个环节分析：
- A → B：这个方向通常怎么走？有没有技术上明显走不通的地方？
- B → C：同理
- C → D：同理

**管家主动给出判断，不要问用户"能不能通"。** 用户不是全知的，问他是把球踢回去。

分析完后呈现给用户：

"我看了你的链路 [A → B → C → D]：
- A→B：没问题，[简述理由]
- B→C：可行，[简述理由]
- C→D：这里可能要注意 [X]，因为 [原因]
整体来看这条链路是通的 / 有个潜在问题需要你确认。"

用户确认或补充后，进入下一步。如果用户说"不对，C→D我想的不是这样的"，按用户说的调整后重新审视。

**这一步的价值**：避免在某个模块深究了半小时技术细节，结果发现上下游根本衔接不上。同时让用户感受到管家在帮他思考，不是在审问他。

### 第六步：风险与难点预判

链路确认通畅后，在分阶段之前，先做一次风险扫描。问用户：

- **"你觉得这里面最难的是什么？"**
- **"哪个模块你最没把握？"**
- **"有没有什么前置条件现在还不满足的？"**

用户回答后，管家用自己的理解补充：
"根据你说的，我判断核心风险是 [X]，建议阶段一优先验证这个点。"

**这一步的价值**：
- 风险高的模块优先做，避免把最难的部分留到最后才发现做不了
- 影响阶段划分——风险点应该尽可能排在早期阶段验证
- 识别前置依赖，避免做到一半卡在非技术阻塞上

用户确认后，进入"确定任务阶段"环节。

---

## 学习/复习分支

### 第二步：明确复习目标和范围

问用户：
- **"你要复习什么？"**（一个项目、一个知识点、一套面试题？）
- **"复习到什么程度算完成？"**（搞懂原理、能复现、能讲清楚？）
- **"这个项目/知识点的环境配置或工具链是什么？"**（IDE、芯片型号、框架版本——方便后续回顾时快速还原环境）

用户确认后，进入下一步。

---

## 文档整理分支

### 第二步：明确整理范围和目标

问用户：
- **"你想整理什么？"**（某个项目的笔记、散落的文档、旧项目归档？）
- **"整理的目标是什么？"**（建索引、去重、归档、总结？）
- **"有没有什么分类偏好？"**（按时间、按技术栈、按项目？）

用户确认后，进入下一步。

---

## 方案预研分支

### 第二步：明确预研目标和范围

问用户：
- **"你想研究什么？"**（对比几款芯片、评估通信方案、调研某个技术方向？）
- **"关注哪些维度？"**（性能、成本、生态、功耗、开发难度？）
- **"预研到什么程度算完成？"**（出对比表格、写推荐方案、做原型验证？）

用户确认后，进入下一步。

### 第三步：确认候选清单

问用户要对比哪些候选：
- 如果用户有明确列表 → 逐项记录
- 如果用户只有一个方向 → 管家可以补充建议候选

用户确认清单后，进入下一步。

### 第四步：拟定输出框架

根据预研类型，建议输出结构：
- **方案对比表** — 各候选在各维度的评分
- **推荐方案** — 选哪个、为什么
- **后续步骤** — 如果要落地，下一步做什么

用户确认格式后，进入下一步。

---

## 个人计划分支

### 第二步：明确计划内容

问用户：
- **"这是一个什么计划？"**（训练计划、读书计划、阶段性目标、习惯养成？）
- **"周期多长？"**（一周、一个月、一个季度？）
- **"怎么算完成？"**（可量化的标准是什么？）

用户确认后，进入下一步。

---

## 确定任务阶段（所有分支通用）— 提案模式

前面已经把效果、模块、范围等信息收集完了。现在管家的任务不是继续问，而是**主动合成并提出方案**。

### 合成并提出阶段方案

管家基于前面收集的所有信息，用自己的判断提出一个完整方案，写成一段话给用户看：

```
根据你说的，我理解你要做的是：

[一段话：把最终效果、模块/范围、用户要达成的状态串起来]

我的建议是分 X 个阶段推进：

阶段一：[阶段名称] — 目标：[具体目标]，交付：[具体产出]
阶段二：[阶段名称] — 目标：[具体目标]，交付：[具体产出]
阶段三：...

如果很简单，也可以不分阶段直接做。

你看这个理解和划分对不对？要调整吗？
```

**关键原则**：
- 管家自己判断分不分阶段、分几个——不把决策压力甩回给用户
- 每个阶段必须有明确的目标和交付物，不能只是"阶段一：做硬件"
- 用户只需说"对""不对，阶段二改成..."——比从零自己想阶段划分轻松得多
- 用户说不分阶段或调整阶段数的，按用户说的改

用户确认（或调整）后，进入下一步。确认下来的阶段划分会直接影响后续文档结构。

---

## 工程项目分支 — 技术选型

分阶段完成后，回到工程项目，逐阶段确定技术栈。**每次只问一个阶段的技术选型，不要跨阶段混杂。**

### 第七步：问技术栈

针对当前讨论的阶段（或整体，如果不分阶段），逐个模块问具体技术选型：

- 用什么技术栈？有哪些核心技术选型需要确定？

每个模块确认后再问下一个。用户说"待定"的先标待定，不追问。

用户回答后，复述确认。

### 第八步：问模块内部架构

逐个模块问工作方式、数据处理、难点。没想好的标待定。

**注意**：如果分了阶段，各阶段的技术选型可能不同（比如早期阶段用快速方案验证，后期阶段换更成熟的技术栈）。问清楚每个阶段各自的技术选型，不要假设全程统一。

---

## 拟定文档清单

读 `库级md文件/库文件模板参考.md`，根据前面收集到的信息，判断本次任务适合哪些文件类型，列出建议清单给用户确认。

判断依据：
- **任务类型**：工程项目 vs 学习复习 vs 文档整理 vs 方案预研 vs 个人计划
- **任务复杂度**：简单任务可以减少文件，复杂任务需要增加
- **用户的习惯**：如果用户之前喜欢写踩坑记录就保留，否则跳过

清单列出后，逐一问用户：
- "我建议生成以下 N 个文件：[列表]。你觉得哪些要、哪些不要？有没有其他想加的？"
- 用户确认后按确认清单生成。

---

## 生成 MD 文档

根据确认的文档清单和阶段选择，生成对应的骨架文件。

工程项目的典型结构（供参考）：

```
任务目录/
|-- 任务介绍.md       - 任务名称、目标、效果描述、模块划分、数据链路图
|-- 开发计划.md       - 步骤清单（初始为空，后续补充）
|-- 学习记录.md       - 空模板
|-- 踩坑记录.md       - 空模板（如果需要）
|-- 技术决策.md       - 空模板（如果需要）
|
|-- 阶段一_XXX/       （分阶段时启用）
|   |-- 阶段介绍.md
|   |-- 阶段进度.md
|
|-- 阶段二_XXX/       （以此类推）
```

学习/复习任务的典型结构（供参考）：

```
任务目录/
|-- 任务介绍.md       - 复习目标、范围、时间安排
|-- 学习笔记.md       - 学到的东西、重点、疑问
|-- 进度跟踪.md       - Checklist 跟踪
```

方案预研任务的典型结构（供参考）：

```
任务目录/
|-- 任务介绍.md       - 预研目标、候选清单、关注维度
|-- 方案对比.md       - 各候选维度的详细对比
|-- 推荐方案.md       - 结论和理由
|
|-- 阶段一_调研/      （分阶段时启用）
|-- 阶段二_验证/
```

个人计划的典型结构（供参考）：

```
任务目录/
|-- 任务介绍.md       - 计划目标、周期、完成标准
|-- 进度跟踪.md       - 各阶段的完成情况
|
|-- 阶段一_XXX/
|-- 阶段二_XXX/
```

每个文件写上标题和基本结构。内容不填满，留出骨架让用户后续补充。

---

## 完成后

生成完毕后告诉用户：
"任务已创建，在 XXX 路径下。你现在可以：跟管家讨论第一步做什么、直接开始执行、或者随时说'换一个任务'切换。"

然后退出任务初始化模式，返回管家状态。
