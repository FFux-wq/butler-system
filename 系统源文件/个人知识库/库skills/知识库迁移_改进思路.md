# 知识库迁移 · 改进思路

> 2026-07-21。迁移到底产出什么？还没想清楚。先记下来，明天实际跑一遍两个版本再看。

## 问题

迁移的核心价值不在"搬文件"——管家扫代码能理清楚的（架构、模块、技术栈），用户自己看一眼目录也能想起来。真正的价值在代码里看不见的东西——调过的坑、选错的方案、以后还能用的模块经验。

但这些东西管家扫不出来，只能靠硬迁移聊天挖出来。聊天能挖多少、挖出来落成什么文件——这两件事现在没想清楚。

## 软迁移

**产出：一个文件 —— 项目介绍.md。**

结构：

```
# XX项目

## 这是什么
一句话 + 最终效果。管家翻到这里知道这个项目值不值得继续看。

## 地图（以后要改这个项目，看这里）
架构、数据流、模块拆解、技术栈、目录结构。
——管家从代码里扫出来，软迁移够用。

## 可复用（以后做类似的东西，看这里）
这个项目做完，哪些东西下次还能用。
——软迁移可能就聊出来一两句。有就有，没有就空着。
```

库级同步更新：项目总结.md、跨项目索引.md。

软迁移没什么争议。快速扫、快速过、以后能翻出来就行。

## 硬迁移

软迁移的一个文件可能兜不住。硬迁移是带着用户把项目从头过一遍——从代码反推架构，聊链路、聊模块、聊技术决策，把代码里看不见的东西挖出来。

聊完之后，产出的可能不只是项目介绍.md。比如：

- 某个模块调了很久，聊出来一堆细节（初始化流程、AT指令格式、固件版本坑）——这些塞进项目介绍的"可复用"段太长，可能需要单独的文件
- 聊的过程中用户想起来当时几个方案对比，最终选了这个——这是技术决策，但跟项目介绍里的事实信息不同
- 聊出来的东西，有些是给"以后改这个项目"用的（架构理解），有些是给"以后做类似东西"用的（模块经验）——可能应该分开

**但具体该产出什么文件、什么格式，现在说不清楚。** 因为还没实际跑过。靠想是想不到的——得真的拿一个老项目迁移一遍，看聊完手上有什么东西、这些东西落到哪些文件里最自然。

## 待验证

明天用 4.0 和 5.0 分别跑一个迁移，实际体验两端：

1. 硬迁移聊完之后，手上到底有哪些类型的信息？
2. 哪些适合塞进项目介绍，哪些撑开了需要单独的文件？
3. 如果有需要单独文件的信息——它的参考模板长什么样？

聊完了再回来改这个文件，把硬迁移的产出定下来。
