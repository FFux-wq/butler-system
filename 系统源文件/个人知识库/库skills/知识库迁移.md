# 知识库迁移

> 库skill 之一。把用户现有的文件（代码、笔记、项目文件夹）导入管家系统，生成初始知识库。

## 使用原则

流程是参考框架，不是剧本。对话中已经覆盖的内容直接跳过，别重复问。只补缺的部分。

## 这个 skill 是干什么的

用户的"知识库"可能是一个项目文件夹、一堆代码、一些散落的 MD 笔记——也可能什么都没有，只有代码仓库。

这个 skill 做的事：
1. 扫描用户指定的路径，识别里面有什么（代码项目、笔记、散落文件）
2. 判断项目状态（已完成还是进行中），根据状态决定处理策略
3. 参考 `库级md文件/库文件模板参考.md`，按管家标准格式生成初始的知识库结构
4. 从代码和文件中提取信息，填充到生成的 MD 文件中
5. 更新根 `.管家/`，注册这个任务

**不做的事**：不修改用户的原始文件（默认复制到新位置），不扫描代码跟知识库对照（那是体检的事），不创建任务级 `.管家/`。

## 触发

- 用户说「帮我把这个项目导进来」「把我之前的代码整理成知识库」「我有一些东西想归到管家管」
- 完整启动时，检测到用户还没有知识库，建议"要不要把你现有的项目导入进来？"

## 流程

### 第一步：确认迁移范围

问用户：
- **你想导入什么？**（一个代码仓库、一个项目文件夹、一堆散落的文件？）
- **路径在哪？**
- **你想复制一份到知识库里，还是直接引用原始路径？**
- **这个项目/文件大概是什么类型的？**（嵌入式 MCU 项目、Python 脚本、Web 项目、学习笔记……）— 如果用户说不清楚，管家直接扫目录结构判断

### 第二步：扫描与判断

这一步分两段：先快速扫一遍判断项目状态，再根据状态深入扫描。

**阶段一：快速扫描，判断状态**

管家先快速扫一眼项目目录，判断它是"已完成"还是"进行中"。不直接问用户——用户可能觉得"说不太清楚"。

判断依据：
- 代码仓库最近有提交吗？看 git log 时间
- 目录里有 TODO、WIP、plan 之类的文件吗？
- 项目文件是否完整——有头有尾 vs 明显还在写
- MD 文档里有没有"待完成""进行中""下一步"这类词
- 纯文档型项目（没有代码）：看文档内容是否有明确的进度标记、未完成项

管家给出判断，告诉用户：
```
我看了一下你的项目，判断它大概是 [已完成/进行中]。理由是：[简述1-2条]。

我按这个状态来规划迁移，如果不对你说一声。
```

用户纠正了再调整。用户没纠正就按这个判断走。

> 为什么要分两条路径：已完成项目走精简归档——生成总结、更新索引就够了，不需要进度跟踪。进行中项目要保留现有 MD 文档、补充缺失、注册到管家工作区，让管家能接手继续推。

**阶段二：深度扫描，按状态调整重点**

根据阶段一的判断，用不同的视角深入扫描目录。

**无论哪种状态，都扫**：
- 项目/任务类型：通过文件后缀和目录结构判断。有 `.c/.h` → 嵌入式 C 项目、有 `package.json` → Node、有 `.py` → Python、纯 MD 文档无代码 → 文档型任务（学习复习、方案预研、个人计划等）
- 目录结构、关键文件清单
- 如果是代码项目：技术栈信息（芯片型号、框架、工具链、语言）
- 如果是纯文档任务：文档结构（几个 MD、各自什么主题、有没有进度/计划/总结类文件）

**如果是已完成项目**，额外关注：
- 项目最终的产出是什么（可执行文件、部署产物、文档……）
- 有项目总结或 README 吗
- 技术栈和模块结构——用于生成项目总结

**如果是进行中项目**，额外关注：
- 现有 MD 文档有哪些（进度记录、更新日志、开发计划、笔记……）
- 哪些是用户手写的，哪些是 AI 写的
- 对照 `库文件模板参考.md`，缺了哪些项目文档
- 当前做到哪一步了（从代码和文档中推断）

### 第三步：列出迁移计划

根据项目状态，给出不同的迁移计划。

**已完成项目的迁移计划模板**：

```
我扫描了你的文件，判断这是一个已完成的 [项目类型] 项目：

- 技术栈/内容：[从文件提取]
- 主要模块/结构：[简述]
- 现有 MD 文件：× 个

我打算：
1. 把项目文件夹复制到知识库 `系统源文件/个人知识库/[项目名]/`
2. 生成项目介绍.md（从代码/文档提取信息）
3. 更新根 .管家/任务队列.md（标记已完成）
4. 更新库级md文件/项目总结.md 和 跨项目索引.md

不创建进度跟踪文件（项目已完结，不需要）。
```

**进行中项目的迁移计划模板**：

```
我扫描了你的文件，判断这是一个进行中的 [项目类型] 项目：

- 技术栈/内容：[从文件提取]
- 当前进度：[从代码和文档推断]
- 现有 MD 文件：× 个（[列出文件名]）
- 缺失的文档：[对照模板列出]

我打算：
1. 把项目文件夹复制到知识库 `系统源文件/个人知识库/[项目名]/`
2. 保留你现有的 MD 文件，内容不动，AI 写的部分加上 butler 标记
3. 补充缺失的文档：[列出要生成的]
4. 更新根 .管家/进度状态.md 和 任务队列.md，注册这个任务

可以开始吗？有要调整的吗？
```

用户确认后进入下一步。用户说不对就改，用户满意就执行。

### 第四步：执行迁移

**复制项目文件**：
- 把整个项目文件夹复制到 `系统源文件/个人知识库/` 下
- 保留原始文件结构，不修改原路径的文件
- 排除编译中间文件（build/、.o、.hex、node_modules/ 等）和大型二进制文件

**已完成项目 — 生成文档**：
- 生成 `项目介绍.md`（项目名、类型、技术栈、模块划分、最终产出）
- 如果用户有现成的 README 或总结，优先用用户的，管家补充缺失信息
- 不生成进度跟踪类文档（开发计划、任务进度等）

**已完成项目 — 更新根 .管家/和库级文件**：
- `任务队列.md`：加到"已完成"区
- `库级md文件/项目总结.md`：新增该项目的条目
- `库级md文件/跨项目索引.md`：新增该项目的技术点和文档索引
- 不更新 `进度状态.md`（已完成项目不需要进度跟踪）

**进行中项目 — 保留现有文档**：
- 用户手写的 MD 文件原封不动复制过去
- 内容中有旧格式标记的，转为 `<!-- butler:managed -->`
- AI 写的内容加上 butler 标记
- 用户手写的内容不加标记

**进行中项目 — 补充缺失文档**：
- 对照 `库文件模板参考.md`，判断哪些文档需要生成
- 从代码扫描结果提取信息，填充：
  - `项目介绍.md`（项目名、类型、技术栈、模块划分、目录结构）
  - 其他缺失的模板文档（骨架，内容标注"待填充"）
- 用户已有的进度记录、更新日志等，不覆盖、不重写
- 管家补充的内容放在 butler 标记区块内，跟用户手写内容分开

**进行中项目 — 更新根 .管家/**：
- `进度状态.md`：在"项目进度"区新增该项目条目，写入当前进度
- `任务队列.md`：加到"进行中"区
- **不创建任务级 `.管家/`**——项目的 MD 文件本身就是项目的工作区，管家状态统一在根 `.管家/` 管理

### 第五步：报告结果

迁移完成后，告诉用户做了什么：

**已完成项目**：
```
导入完成。[项目名] 已归档到知识库。

生成了：
- 项目介绍.md（从代码/文件提取了信息）

更新了：
- 根 .管家/任务队列.md（标记已完成）
- 库级md文件/项目总结.md
- 库级md文件/跨项目索引.md

你现在可以跑一次体检，让管家进一步梳理和校对索引。
```

**进行中项目**：
```
导入完成。[项目名] 已加入知识库。

保留了你的原有文件：
- [列出保留的 MD 文件]

新生成/补充了：
- [列出新生成的文件]

更新了：
- 根 .管家/进度状态.md
- 根 .管家/任务队列.md

你现在可以：
1. 让管家读一下项目介绍，从当前进度继续推进
2. 跑一次体检，进一步梳理索引和补充遗漏
3. 直接在管家对话里开始工作
```

---

## 注意事项

- 不要在用户没确认前执行任何写入操作
- 原始路径的文件不要动——默认复制到知识库目录下
- 从代码提取的信息标注"从代码推断，请确认准确性"
- 用户说不对的直接删或改，不保留
- **不创建任务级 `.管家/`**——项目的 MD 文件就是项目的工作区
- 用户已有的 MD 文档优先于模板——不覆盖、不重写
