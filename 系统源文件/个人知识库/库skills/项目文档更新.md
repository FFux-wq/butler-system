# 项目文档更新

> 库skill 之一。项目进行中维护项目文档。三种方式：你说"帮我记一下"直接补、扫代码发现空白、或推进阶段时自动同步进度。

## 写入规则（所有模式通用）

所有 AI 写入的文档内容，必须用 HTML 注释标记管理区间：

```
<!-- butler:managed id=<唯一id> -->
（AI 管理的区块内容）
<!-- butler:managed-end id=<唯一id> -->
```

- 标记外的内容（用户手写部分），AI 不修改
- 追加新条目时，在已有标记之后新建一个标记区块
- 追加不覆盖已有内容

## 触发

- 用户说「学了个新东西，帮我记一下」「更新一下文档」「加个新模块」时 → 模式A
- 用户说「扫一下代码看有什么没记的」「看看有什么遗漏」时 → 模式B
- 用户说「进展到哪了」「我做完这一步了」「推进一下」「更新进度」时 → 模式C

## 更新原则

日常推进和架构变动是两码事，走不同的更新路径。

### 日常推进（模式C）
只更新：
- `.管家/进度状态.md` — 当前在做什么、进度百分比
- `.管家/任务队列.md` — Checklist 勾选、调整优先级
- 项目的 `开发计划.md` — 已完成步骤标记

不动：
- 项目介绍.md（链路没变就不动）
- 库级汇总文件（跨项目索引、项目总结——留给体检）

### 架构变动（模式A-变体）
数据链路变了（增删模块、换芯片、技术栈调整等），同步更新：
- 项目介绍.md（更新数据链路、模块列表、技术栈）
- 开发计划.md（调整阶段计划）
- .管家/进度状态 + 任务队列

**判断标准**：链路变了就同步，链路没变就不动。

## 模式A：直接更新

### 第一步：判断更新类型

问用户更新什么，然后判断属于哪一类：

**日常记录类**（学了个新东西、记个踩坑、记个决策）→ 直接进入第二步
**架构变动类**（换模块、换芯片、链路调整）→ 走架构变动流程

### 第二步：定位更新目标

确定更新哪个项目和哪个具体文件，然后读取该文件当前内容。读取后再确认用户要更新什么。

### 第三步：读取现有文档

打开对应的 MD 文件，读当前内容。

### 第四步：更新内容

根据用户描述，补充或修改对应章节。不删用户原来写的内容。

### 第五步：确认

把更新后的内容展示给用户。用户确认后写入。用户说不对就改，用户满意就保存。

如果是架构变动，同步更新 .管家/ 工作区和开发计划。

## 模式B：扫代码发现空白

### 第一步：读项目文件

先读项目的关键文件了解全貌：项目介绍.md、项目上下文.md、开发计划.md。

读完后，从 项目上下文.md 末尾的「待补充的 MD 文件」区块中确认哪些空文件是刻意留空的，避免重复提醒。

### 第二步：扫代码目录

遍历项目的源代码目录（`.c` `.h` `.py` 等），识别：

- **芯片/模块**：代码中出现的芯片型号、模块名称
- **通信协议/外设**：USART、SPI、I2C、DMA 等，以及特殊配置模式
- **架构模式**：分层、状态机、事件驱动等
- **工具链/编译环境**：编译器、特殊选项、编码问题

### 第三步：对照 MD 文件

| 代码中有 | 对照的 MD 文件 | 检查什么 |
|----------|---------------|---------|
| 芯片/模块 | 单模块总结/ | 有没有对应的总结文档 |
| 新外设/协议 | 学习记录.md | 学了没记 |
| 架构模式 | 技术决策.md | 做了决策没写 |
| 工具/编译 | 工具调用备忘.md | 用了新工具没记 |
| Bug | 任务规划.md 的 Bug 清单 | 修了没记录 |
| 跟项目介绍不符 | 项目介绍.md | 技术栈变了没更新 |

### 第四步：列出空白清单

按两类汇总：「代码里有但 MD 没记录」和「MD 文件本身的问题」。用问句呈现，不直接写入。

### 第五步：用户确认后写入

用户选择要补什么，进入模式A的写入流程（第二步起）。遵循写入规则中的标记约定。

## 模式C：阶段推进同步

### 第一步：确认进展

用户说"进展到哪了""我做完了这步""推进一下"时，确认当前完成的内容。

### 第二步：更新工作区和开发计划

- 更新 `.管家/进度状态.md` 中的进度百分比和当前任务
- 更新 `.管家/任务队列.md` 中的 Checklist
- 更新项目的 `开发计划.md`，已完成步骤打勾

### 第三步：判断是否需要架构更新

问用户一句："这次推进有没有涉及架构变动（模块增减、技术栈变化、数据链路调整）？"

- 没有 → 结束，告知用户当前进度
- 有 → 切换到模式A的架构变动流程，同步更新项目介绍.md
