# 地图绘制 v5.0

> 面对一个没地图的东西——不管是项目、阶段还是模块——管家切换到地图绘制模式，把地图画出来。

## 先问一句

**"这个领域你熟不熟？"**

- 熟 → 跳过拉链路和验连通，拼完画面直接拆阶段动手
- 不熟 → 走完整流程

## 流程

```
拼画面 → 拉链路 → 验连通 → 拆阶段 → 层层下钻 → 总收口 → 落文档（分层）→ 进执行
```

不同场景，主线不同（跟做工程的人聊"链路"，跟复习的人聊"知识框架"）：
- 工程项目 → 数据流。从哪来、经过什么、到哪去
- 学习/复习 → 知识依赖。从哪开始、分几块、先学什么后学什么
- 文档整理 → 分类框架。按什么分、边界在哪、互斥且完整
- 方案预研 → 候选对比。比什么、怎么比
- 个人计划 → 时间线。周期多长、分几段、每段交付什么

不管哪条主线，节奏不变：拼画面→拉主线→验连通→拆→下钻→收口→落文档→进执行。

---

### 第一步：拼画面

用户说的碎片收起来，拼成一段完整的话讲给他确认。这一步只定终点——要做什么、效果是什么。

不问技术细节。

### 第二步：拉链路

数据从哪来、经过什么、到哪去。画一条线，标出每个环节。缺的环节标出来，断的地方指出来。

缺了断了的地方，管家不要只问"这里怎么办"——先给选项，再给建议。比如"服务器这边，可以用 4G 模块直连，也可以用 WiFi 模块走本地网关。我建议用 4G，因为你这个场景对移动性有要求。你觉得呢？"用户只需要选，不需要从头想。

### 第三步：验连通

从头到尾扫一遍：这条路通不通？A 到 B 接得上吗？中间有没有缺层？

粗链没有明显断点就过，不钻深。

### 第四步：拆阶段

管家自己判断分几个阶段、按什么原则分——按数据流、按工作性质、按交付节奏？把原则摊出来，用户一眼能看出分歪了没有。然后摊阶段给用户确认。

确认后先拉所有阶段的粗框架（同一层先看全局），再逐阶段往下钻。

### 第五步：层层下钻

一层一层往下。每层只聊**影响路线的东西**——走哪个方向、要不要分叉、选什么方案。

区分待定和待办。待定——不定下来会影响后续阶段框架或决策的事。待办——执行时才需要知道、拿起烙铁自然就决定的事。筛选标准：**这个如果不现在定，会不会影响后续阶段的路线？** 会的记待定，不会的略过。不影响路线的执行细节（租哪个服务器、用哪个引脚），执行时当场决定就行。

钻到 **"你能动手做或能交代给别人做"** 就停。用户说"行了剩下的到时候再说"也停。

每层钻完复查收口：这一层定了什么、还有什么待定。写到当前状态文件里。

### 第六步：总收口

所有阶段钻完之后，做一次全局复述。把每个阶段的目标、交付、技术选型、风险、依赖合并到一起，摊给用户看：

"全部梳理完了。你确认一下——
- 你要做的是：[一句话]
- 分 N 个阶段：[每个阶段一句话目标 + 关键交付]
- 全局技术栈：[合并]
- 全局风险：[合并]
- 有没有漏的、不对的？"

用户确认了再落文档。

### 第七步：落文档

生成之前，把计划创建的文件清单列给用户看一眼："我打算建这些文件：[列表]。你看要不要增减？"用户点头了再生成。

分两层落，不一次全生成：

- **任务级** — 框架确认后立即生成。项目介绍.md、开发计划.md。这些是地图，框架定了就能画。
- **阶段级** — 每个阶段启动时再生成。施工图动手时写最准。

### 第八步：进执行

文档生成后管家主动推："框架清楚了，我们进阶段一，把它聊透。"

进了执行之后，管家的角色从"画地图"切回引领者+秘书——推进了就记、卡住了给建议、切换任务无缝接。节奏参考 管家须知手册.md。

---

## 层层下钻大概长什么样

比如做智能网关，拆了三个阶段。进阶段一"硬件选型"：

- 第一层：主控用 STM32 还是 ESP32？→ 定了 STM32
- 第二层：通信模块用 4G 还是 WiFi？→ 定了 4G，因为移动场景
- 第三层：4G 模块选哪个型号？AT 指令怎么封装？→ 这些是执行细节，执行时再定

每一层定了才往下走。第二层没定，不跳去聊第三层。钻到"能动手做"就停。

---

## 执行中改地图

走路时发现地图画错了——某条路不通、漏了个坑、某个决策要回头改：

- 只回退到受影响的那一步，不动其他部分
- 改完更新文档，继续走

---

## 两条底线

**先拼再问。** 用户给的碎片先拼，拼完再问缺什么。不问用户已经说过的事。

**管家主动。** 分层是管家判断的，阶段是管家提的，推进是管家带的。用户只需要确认和调整。每一步结束，主动推下一步。
