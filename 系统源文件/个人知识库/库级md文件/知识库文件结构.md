# 知识库文件结构

> 一个参考——让用户和管家大概知道知识库长什么样。不是规范，管家根据实际情况自己判断。

## 整体大概长这样

```
个人知识库/
├── .管家/              管家自己的工作区
├── 库skills/           封装好的工具流程
├── 库级md文件/         跨项目共享的参考文件
├── 想法池.md            还没归属的想法
└── [项目名]/           一个个项目
```

| 放哪 | 放了些什么 | 平时谁在管 |
|------|-----------|-----------|
| `.管家/` | 管家身份、须知手册、进度状态、任务队列、时间线、用户偏好、可用工具卡 | 管家 |
| `库skills/` | 地图绘制、项目文档更新、项目归档、知识库体检、知识库迁移 | 系统 |
| `库级md文件/` | 模板参考、文件结构、用户画像、跨项目索引、全局地图、项目总结、模块芯片理解 | 体检/归档 skill |
| `想法池.md` | 还没想好放哪的想法——可能变成一个新项目，也可能哪天归到某个项目里去 | 管家 + 用户 |

## 一个项目目录里通常有什么

画地图的时候管家自己判断。不同类型的项目建的东西不一样——嵌入式项目可能有模块总结，纯软件项目可能没有。大概这些：

**起项目时通常会建：**

- `项目介绍.md` — 这个项目是什么、怎么做、分几块、用什么
- `开发计划.md` — 分几个阶段、每个阶段做什么、当前在哪

**推进过程中，碰到场景了就建：**

- `项目由来.md` — 为什么做这个项目
- `学习记录.md` — 学到新东西了记一笔
- `踩坑记录.md` — 遇到坑、爬出来了，记下来
- `测试记录.md` — 有测试验证的时候
- `技术决策.md` — 做了重要的取舍
- `工具调用备忘.md` — 项目特殊的环境或工具用法
- `部署发布说明.md` — 需要部署到什么地方
- `代码评价.md` — 阶段性回头看一下代码质量
- `改进思路.md` — 推进过程中冒出来的"这个可以更好"，不急着做，先记着
- `单模块总结/` — 有重要的模块需要单独总结（是个目录）
- `项目上下文.md` — 管家自己维护的对话快照

**分了阶段的项目，每个阶段目录里通常有：**

- `任务介绍.md` — 这个阶段的目标、管哪些模块、输入输出
- `任务进度.md` — 阶段里的步骤清单、卡在哪

硬件项目打样的时候可能还有个 `BOM清单.md`。

## 想法放在哪

两层，各管各的：

- **还不知道归哪的想法** → 根目录的 `想法池.md`。可能是未来的项目，也可能是还没想清楚方向的念头
- **明确挂在某个项目下面的改进思路** → 那个项目目录里的 `改进思路.md`。比如"WiFi 连接改非阻塞""MQTT 加重连机制"——它属于这个项目，但还不急着做

什么时候想法池里的东西成熟了，管家走地图绘制，它就变成了一个真正的项目目录。
